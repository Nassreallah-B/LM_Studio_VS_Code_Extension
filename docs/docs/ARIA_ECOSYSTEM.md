# ARIA Agent Ecosystem — Advanced Orchestration

## 1. Vision & Architecture

The ARIA (Admin Runtime Intelligence Assistant) ecosystem is an **Event-Driven Orchestration** model. Instead of a single generalist assistant, it utilizes a core of specialized agents coordinated by a Lead Architect. This ensures high-precision interventions in complex areas like Database Security, RTL UI, and Large-scale Refactoring.

### Architecture Layers

```text
┌───────────────────────────────────────────────────────────┐
│                     User Request                          │
└─────────────────────────┬─────────────────────────────────┘
                          ▼
```

## 3. SPARC Integration (Sense → Plan → Act → Reflect → Correct)

### Automated Analysis on Orchestrator Tasks

When an `aria-orchestrator` agent starts a task, the system **automatically** runs SPARC analysis on round 1:

1. **Sense** — Analyzes the user's prompt to detect:
   - Impacted domains (database, security, ui, api, testing, performance, refactoring)
   - Task complexity (low / medium / high)
   - Risks (destructive_operation, schema_change, production_impact, security_sensitive)
   - Prior learned patterns from MemoryDB

2. **Plan** — Generates a structured execution plan:
   - Subtask decomposition with agent assignments
   - Topology recommendation (pipeline / hub-spoke / map-reduce)
   - Risk mitigations
   - Step budget per subtask

3. **Context Injection** — The analysis is injected into the conversation as a structured block:

```text
[SPARC Analysis — Auto-generated by Sense+Plan]
Complexity: medium
Domains: database (→ database-expert, high), security (→ security-sentinel, critical)
Risks: schema_change, security_sensitive
Mitigations:
  schema_change: Create a rollback migration alongside the new migration.
  security_sensitive: Mandatory security-sentinel audit with PASS verdict before merge.
Suggested topology: pipeline
Subtask plan:
  [task_1] database → database-expert (priority: high, budget: 18 steps)
  [task_2] security → security-sentinel (priority: critical, budget: 23 steps)
Estimated total steps: 41

Use this analysis to guide your orchestration strategy.
```

### Safety
- SPARC analysis is **non-blocking** — if it fails, the task continues normally
- Only activates for `aria-orchestrator` agents on round 1
- State is persisted to MemoryDB for post-task auditing

## 4. MutationGuard — Role-Based Access Control

Each agent type has explicit permissions for write, shell, and delete operations:

| Agent | Write | Shell | Delete | Notes |
|---|---|---|---|---|
| `aria-orchestrator` | ❌ | ❌ | ❌ | Delegates only |
| `general-purpose` | ✅ | ✅ | ❌ | Standard worker |
| `database-expert` | ❌ | ✅ (approval) | ❌ | SQL queries need approval |
| `security-sentinel` | ❌ | ✅ | ❌ | Audit only |
| `refactoring-expert` | ✅ | ✅ | ❌ | Full edit access |
| `Explore` / `Plan` | ❌ | ❌ | ❌ | Read-only |
| `verification` | ❌ | ❌ | ❌ | Read-only |

**Blocked paths** (all roles): `.env`, `package-lock.json`, `node_modules/`, `.git/`

## 5. The "Always-On" Philosophy

Agents operate under strict operational protocols ensuring production safety:

### A. Mandatory Verdicts

Agents specialized in Security and Verification MUST conclude their tasks with one of:

- `[VERDICT: PASS]` — Clean bill of health.
- `[VERDICT: FAIL]` — Critical blockers or vulnerabilities detected.
- `[VERDICT: PARTIAL]` — Changes are safe but require non-critical follow-up.

### B. Specialized Step Budgets (Rounds)

Complexity varies by domain:

- **Refactoring tasks** are allocated up to **80 rounds**
- **Orchestration** is allocated **50 rounds** for multi-agent coordination
- **Standard tasks** remain at the default 6 rounds (configurable)

### C. "Premium" UI Standards

Any agent touching the UI is programmed to enforce CloudZIR premium standards:

- **Typography:** Outfit (Google Fonts)
- **Styling:** Glassmorphism, smooth gradients (HSL based), vibrant color palettes
- **RTL Fluidity:** Native support for Arabic (mirrored layouts) and seamless switching

## 6. Swarm Topologies

For multi-domain tasks, the system supports three orchestration topologies:

| Topology | Description | When Used |
|---|---|---|
| **Pipeline** | A → B → C sequential | Tasks with strict dependencies |
| **Hub-Spoke** | Central coordinator + parallel agents | Independent multi-domain work |
| **Map-Reduce** | Parallel mappers → single reducer | Distributed analysis + synthesis |

Topology is recommended by SPARC analysis but the orchestrator makes the final decision.

## 7. Operational Workflows

### The Trigger Matrix

- **Schema/RLS Changes:** Requires `database-expert` + `security-sentinel`
- **UI Tweaks:** Requires `rtl-ui-auditor`
- **Bug Fixes:** Requires `general-purpose` → `verification`
- **Large Refactoring:** Handled by `refactoring-expert` after a `Plan` phase
- **Design System:** Handled by `design-system-expert` (Plugin bridge)
- **Multi-Domain Tasks:** SPARC decomposes → appropriate topology applied

## 8. Persistence & Audit Trail

All agent activity is persisted locally:

| Data | Primary Store | Secondary Store |
|---|---|---|
| Events | `agent-runtime/events.json` | `MemoryDB.events` table |
| Agent State | `agent-runtime/agents/*.json` | `MemoryDB.agent_memory` table |
| SPARC Analysis | — | `MemoryDB.workflow_state` table |
| LLM Usage | `agent-runtime/costs.json` | `MemoryDB.performance_metrics` table |
| Audit Log | `agent-runtime/logs/audit.log` | — |

**Everything is stored locally** — no external databases, no cloud storage.
